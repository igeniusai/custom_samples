# TLS Scripts

Bash scripts for generating self-signed TLS certificates and loading them into a Kubernetes cluster as `kubernetes.io/tls` Secrets.

> These certificates are intended for local development only. Do not use in production.

## Prerequisites

- `openssl` — certificate generation (`brew install openssl`)
- `kubectl` — cluster operations, configured to point at the target cluster

## Directory structure

```text
scripts/tls/
├── create_certs.sh   # Generate a CA and a signed server certificate
├── cluster_tls.sh    # Load a certificate pair into the cluster as a TLS Secret
├── Taskfile.yaml     # Task runner shortcuts
└── certs/            # Output directory for generated files (git-ignored)
    ├── ca.key        # CA private key   — keep secret
    ├── ca.crt        # CA certificate   — import into browsers/OS to establish trust
    ├── <name>.key    # Server private key
    ├── <name>.csr    # Certificate signing request (intermediate, can be discarded)
    └── <name>.crt    # Signed server certificate
```

## Quickstart

```bash
# 1. Generate the CA and server certificate
./create_certs.sh --cert-name myservice --cert-cn myservice.local --cert-san DNS:myservice.local

# 2. Trust the CA on macOS
sudo security add-trusted-cert -d -r trustRoot \
  -k /Library/Keychains/System.keychain \
  ./certs/ca.crt

# 3. Add a hosts entry so the hostname resolves locally
echo "127.0.0.1 myservice.local" | sudo tee -a /etc/hosts

# 4. Push the secret to the cluster
./cluster_tls.sh --cert-name myservice --secret-name myservice-tls --namespace test
```

## `create_certs.sh`

Generates a local CA and a server certificate signed by that CA. All output files are written to `certs/`.

```text
Usage: ./create_certs.sh [OPTIONS]

Options:
  --ca-cn     CN for the CA certificate        (default: TestLocalCA)
  --ca-days   CA cert validity in days         (default: 1826  ≈ 5 years)
  --cert-name Output file prefix               (default: server_cert)
  --cert-cn   CN for the server certificate    (default: test.local)
  --cert-days Server cert validity in days     (default: 825  ≈ 2 years)
  --cert-san  Subject Alternative Names        (default: DNS:test.local)
  --key-size  RSA key size in bits             (default: 2048)
```

### Examples

```bash
# Use all defaults
./create_certs.sh

# Custom hostname
./create_certs.sh \
  --cert-name myservice \
  --cert-cn myservice.local \
  --cert-san DNS:myservice.local

# Multiple SANs (wildcard + localhost + loopback IP)
./create_certs.sh --cert-san 'DNS:*.local,DNS:localhost,IP:127.0.0.1'

# Custom CA with shorter validity
./create_certs.sh --ca-cn MyDevCA --ca-days 365 --cert-days 90
```

### Output files

| File | Description |
| --- | --- |
| `certs/ca.key` | CA private key — keep this secret |
| `certs/ca.crt` | CA certificate — import into macOS/browsers to trust the server cert |
| `certs/<name>.key` | Server private key |
| `certs/<name>.csr` | Signing request — intermediate file, safe to discard |
| `certs/<name>.crt` | Signed server certificate |

## `cluster_tls.sh`

Deletes (if present) and recreates a `kubernetes.io/tls` Secret in the target namespace using a certificate pair generated by `create_certs.sh`. The `--cert-name` flag must match the one used in `create_certs.sh`.

```text
Usage: ./cluster_tls.sh [OPTIONS]

Options:
  --namespace    Kubernetes namespace to create the secret in  (default: test)
  --secret-name  Name of the Kubernetes TLS secret             (default: tls-secret)
  --cert-name    File prefix of the cert/key pair to load      (default: server_cert)
```

### Usage examples

```bash
# Use all defaults
./cluster_tls.sh

# Match the cert name used in create_certs.sh
./cluster_tls.sh \
  --cert-name myservice \
  --secret-name myservice-tls \
  --namespace prod
```

## Trusting the CA on macOS

Import the CA into the System keychain so that browsers and OS tools accept the server certificate:

```bash
sudo security add-trusted-cert -d -r trustRoot \
  -k /Library/Keychains/System.keychain \
  ./certs/ca.crt
```

After importing:

1. Open **Keychain Access → System → Certificates**, find the CA by its CN (e.g. `TestLocalCA`).
2. Double-click it → expand **Trust** → confirm **SSL** is set to *Always Trust*.
3. **Fully quit and reopen** your browser (Cmd+Q) — browsers cache the trust store and must be restarted to pick up changes.

## Verifying the certificate in use

Check that the Ingress is serving your certificate and not the nginx fallback:

```bash
openssl s_client -connect 127.0.0.1:443 -servername test.local 2>/dev/null \
  | openssl x509 -noout -text \
  | grep -A2 "Issuer\|Subject\|DNS"
```

The `Issuer` field must match your CA CN (e.g. `CN=TestLocalCA`). If it shows `Kubernetes Ingress Controller Fake Certificate`, the Secret is missing or in the wrong namespace — verify with:

```bash
kubectl get secret tls-secret -n test
kubectl describe ingress -n test
```
